### The Complete Language Development Ecosystem
Think of language software like a city's infrastructure. You don't need every building, but you need the right buildings in the right places, connected by roads that actually work.
### Phase 1: Foundation Building
**Primary Activities:** Word collection, phonological analysis, orthography development
**Key Tools:**
- **Word Collection:** The Combine (rapid), WeSay (community-friendly), FieldWorks Lite (streamlined)
- **Sound Analysis:** Phonology Assistant, Speech Analyzer
- **Documentation:** SayMore
**Why This Order Matters:** You can't build a stable writing system without understanding sounds. You can't do quality translation without a stable writing system. Teams that skip this spend years fixing problems.
**Lesson:** A West African team started translating before finalizing orthography. After 2 years and completing Matthew, the community changed how they represented 3 sounds. Result? Two years of rework. Invest time in the foundation.
### Phase 2: Parallel Development
Once orthography is stable, three tracks run simultaneously:
**Track 1: Lexicon**
- **Core:** FLEx (linguistic powerhouse) or FieldWorks Lite (simpler alternative)
- **Publishing:** FLEx → Lexique Pro / Dictionary App Builder / Webonary
**Track 2: Translation**
- **Drafting:** Adapt It (re-use translations), FLExTrans (from lexicon), AI tools
- **Collaboration:** Paratext (industry standard), Scripture Forge
- **Checking:** Paratext checks, Transcelerator, Community Checking App
- **Publishing:** PTXprint, Scripture App Builder, Publishing Assistant
**Track 3: Literacy**
- **Primer:** PrimerPro
- **Materials:** Bloom (books, multiple formats)
- **Distribution:** Bloom Reader, Reading App Builder
**Plus: Oral Bible**
- **Dramatization:** Glyssen
- **Recording:** HearThis, Render
- **Story Management:** One Story Editor
- **Video:** Story Producer
### Understanding Data Flows: The Integration Secret
**Key Data Pathways:**
**Lexicon:** WeSay/Combine → \[LIFT\] → FLEx → \[SFM/LIFT\] → Publishing tools
**Translation:** First Draft → \[USFM\] → Paratext → \[USFM\] → Publishing tools
**Literacy:** Orthography → PrimerPro → Bloom → Distribution
**Critical File Formats:**
- **USFM** - Paratext's Scripture format
- **LIFT** - Lexicon exchange between tools
- **SFM** - General linguistic data
**When teams ask:** "Can we move data from Tool A to Tool B?" → Check if they share a common file format.
### Tool Selection Framework
### Tool Selection Framework
**Ask These Questions:**
1. What's the team's primary goal right now?
2. Who will use this tool? (Technical level?)
3. What data do they already have?
4. What's their long-term vision?
**Quick Examples:**
- Community lexicon work → WeSay
- Rapid word collection → The Combine
- Morphology analysis needed → FLEx
- Related language exists → Adapt It
- Literacy materials → Bloom
### The 4-Step Troubleshooting Method
**1. CLARIFY**
- What exactly happens vs. what should happen?
- When did it start? What changed?
- Can they reproduce it?
**2. CATEGORIZE**
- User knowledge gap?
- Workflow issue?
- Configuration problem?
- Software bug?
- Data corruption?
**3. CHECK**
- Official documentation
- Community forums
- Known issues lists
- Your knowledge base
**4. CHOOSE**
- **Solve directly** - You know the answer, low-risk fix
- **Point to resources** - Good documentation exists
- **Escalate** - Data loss, bugs, or beyond your expertise
- **Defer** - Need to research first
**Golden Rule:** Never guess if you're unsure. Say "I don't know, but I'll find out."
### Key Takeaways
1. **Language development is phased:** Foundation first (phonology, orthography), then parallel work (lexicon, translation, literacy)
2. **Tools are interconnected:** Data flows through standard formats (USFM, LIFT, SFM)
3. **Know the hubs:** Paratext (translation), FLEx (linguistics), Bloom (literacy)
4. **Your role:** Guide teams to the right tools; you don't need to be an expert in everything
5. **Troubleshooting:** Use the 4-step method: Clarify, Categorize, Check, Choose
